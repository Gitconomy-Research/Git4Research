# Gitconomy Research社区文档规范指南

## 1. 宗旨与目的

在`人人都是研究员`的Gitconomy开放研究社区中，让每一次`Commit`都能被社区快速阅读、复现、再创造。本指南统一文档风格、目录结构、元数据与提交流程，确保人类与 AI 都能无缝协作。

## 2. 适用范围

本指南适用于所有向Gitconomy Research所有仓库提交的文档类内容，包括但不限于：

- 研究论文、报告解读
- 项目白皮书、技术文档
- 教程、工作流程指南
- 会议记录、社区治理提案
- README 文件
-  任何其他以 .md, .pdf, .txt 等格式存储的说明性文件。

## 3. 文档规范指导思想

本部分旨在阐明本指南背后的“原因”，将文档定位为开源环境下开放研究流程中不可或缺的一环，而不仅仅是一项繁琐的任务。

在开放式研究的背景下，文档是建立信任和确保科学有效性的首要工具。它不仅是项目成果的说明书，更是研究过程透明度和可信度的载体。传统的开源软件文档侧重于可用性和功能实现，而开放式 研究 文档则必须将可复现性置于核心地位。一份高质量的文档，意味着研究成果是可被验证、可被信赖的，从而使其他研究者能够在此基础上进行验证和扩展 。因此，本社区的文档不仅要服务于代码的使用者，更要服务于科学的审查者。

一个成功的开源研究社区，其本质在于融合两种截然不同的文化：软件开发的敏捷迭代文化与学术研究的严谨验证文化。前者追求快速原型和功能交付，文档侧重于“快速上手”和API参考 ；后者则强调同行评审和结果验证，文档侧重于详尽的方法论、数据来源和理论依据。若无法有效弥合这两种文化的差异，社区将面临两种潜在的失败模式：其一，项目因缺乏严谨的方法学和数据文档而科学性不足，无法获得学术界的认可；其二，项目因文档过于晦涩、理论性过强而吓退潜在的社区贡献者，导致项目停滞不前。

因此，本指南的核心目标是建立一种`混合文档模型`。它既要提供顶级开源软件项目所具备的清晰、简洁的快速入门指南和用法示例 ，又要强制执行学术研究所需的严格的数据、代码和实验方法文档标准。这种模型的建立，旨在将文档本身作为澄清思路、帮助他人理解期望的工具 ，并最终将严谨的科学数据管理融入社区工作流程，使其成为迈向可复现研究的第一步 。通过这种方式，文档将成为连接敏捷协作与科学严谨性的桥梁，从而构成本社区独特的价值主张。

Gitconomy Research开放研究社区文档规范的基本原则：

1. `乐高原则`——把文档拆成最小可复用单元（数据说明、方法描述、代码片段、思维笔记），像乐高一样随取随拼。
2. `AI 友好`——所有文档须同时满足“人类可读 + LLM 可解析”。使用显式标题、列表、代码围栏，避免隐式引用。
3. `一次写作、多处复用`——任何文档块（如方法描述）应能在 README、论文、教学课件中无缝引用，借助 Markdown 的 transclusion 与变量替换实现。

## 4. 文档目录结构

### 4.1 文档夹命名规范

- 请使用小写字母
	- *示例：open-research*
- 单词间使用中划线连字符（-）分隔
- 使用数字前缀给文件夹排序，方便 GitHub/GitCode Web 界面浏览
	- *示例：*

~~~
📁 Git4Research
├── 📄 README.md              
├── 📄 CONTRIBUTING.md        
├── 📁01-open-perspetives/
├── 📁02-open-research/
├── 📁03-open-thinking/
├── 📁05-open-education/
~~~

- 名称应简洁明了，反映内容主题
	- *示例：ai-ethics, open-governance, digital-commons*

## 4.2 文档目录规范

1. `研究类文档`

~~~
📁 research-topic/
├── 📁 01-papers/       # 完整论文或报
├── 📁 02-data/         # 研究数据
│   ├── raw/            # 原始数据
│   └── processed/      # 处理后的数据
├── 📁 03-methods/      # 研究方法描述
├── 📁 04-analysis/     # 分析代码和结果
├── 📁 05-assets/       # 图表和可视
└── 📄 README.md        # 研究概述和导航
~~~

2. `教育类文档`

~~~
📁 education-topic/
├── 📁 01-introduction/    # 课程说明
├── 📁 02-syllabus/        # 教学大纲
├── 📁 03-lessons/         # 课程内容
│   ├── 📄 lesson-01.md
│   └── ...
├── 📁 04-exercises/       # 练习和实践项目
├── 📁 05-resources/       # 学习资源
└── 📄 README.md           # 课程概述和导航
~~~

3. `评论/思想类文档`

~~~
📁 perspective-topic/
├──  📁 01-case-studies/    # 案例研究
├──  📁 02-references/      # 参考资料
├── 📄 README.md            # 主题概述和导航
├── 📄 core-concepts.md     # 核心概念阐述
└── 📄 pplications.md       # 应用场景
~~~

## 5. 通用写作与格式标准

### 5.1 命名规范

1. `文件命名`

- 中英文即可，如果文件名为英文，请使用小写字母
	- *示例：基于“贡献即要素”的社区角色重构思考.md*
- 单词间使用下划线连字符（_）分隔
	- *示例：syllabus_01.md*
- 如果包含版本信息时使用 v1, v2 等后缀
- 包含日期信息（格式：YYYYMMDD）

2. `图片命名`

- 使用描述性名称
	*示例：gitconomy_mvr_framework.svg*

### 5.2 文档格式规范

所有文档应使用 Markdown 格式编写，遵循以下规范：

1. `标题层级`

- 文档标题使用一级标题（# 标题）
- 主要章节使用二级标题（## 章节）
- 子章节使用三级标题（### 子章节）
- 最多使用到四级标题（#### 小节）

2. `列表格式`

- 无序列表使用连字符（-）
- 有序列表使用数字加点（1.）
- 列表缩进使用两个空格

3. `代码块`

- 使用三个反引号标记代码块
- 指定语言类型以启用语法高亮
- `示例：

```Python
def example_function():
    return "This is a code example"
```

4. `表格`

- 使用标准Markdown表格语法：
- 表格前后留一空行
	- *示例：*

|列名1|列名2|列名3|
|:---|:---|:---|
|数据1|数据2|数据3|

5. `图片引用`

- 使用相对路径引用仓库内图片
- 提供清晰的alt 文本，兼顾无障碍与 LLM 理解。
	- 示例：\!\[GItconomy社区所有制研究计划概览](./assets/gitconomy_mvr_framework.svg)

### 5.3 内容规范

1. `元数据`

对于较长的研究文档或文章，建议在 Markdown 文件顶部添加 YAML Front Matter 来提供元数据，便于管理和检索。元数据可以采用显式或隐式方式：

**显式元数据（推荐用于正式研究文档）**

文档开头使用YAML格式的元数据块：

~~~
---
title: 文档标题
author: 作者名称
date: YYYY-MM-DD
version: x.y.z
category: [研究/思想/教育]
tags: [标签1, 标签2]
---
~~~

**隐式元数据（适用于简短文档或非正式贡献）**

在文档正文中自然融入必要信息：

~~~
# 文档标题

作者：作者名称 | 日期：YYYY-MM-DD | 版本：x.y.z

[文档正文开始...]
~~~

对于简短或临时性文档，甚至可以只在文档标题下方添加作者和日期信息：

~~~
# 文档标题

作者：作者名称
日期：YYYY-MM-DD

[文档正文开始...]
~~~

无论采用哪种方式，都应确保包含基本的作者和日期信息，以便追踪文档来源和版本。

2. `摘要`

研究类文档应在元数据后提供不超过200字的摘要，概述文档的核心内容和贡献。

3. `引用格式`

- 学术引用使用APA格式
- 引用应在文档末尾的"参考文献"部分列出
- 引用编号在正文中使用方括号标注，如 [1]

4. `术语使用`

- 首次出现的专业术语应提供简短解释
- 可考虑在文档末尾添加术语表
- 保持术语使用的一致性

5. `多语言支持`

**文件命名**

- 多语言版本的文档应在文件名中添加语言代码
	- 示例：contribution-guidelines-zh.md（中文版），contribution-guidelines-en.md（英文版）

**目录组织**

- 如有必要，可为不同语言版本创建单独的目录
	- 示例：docs/zh/, docs/en/

6. `翻译管理`

- 原始语言文档修改后，应更新所有翻译版本
- 翻译文档应在元数据中注明原文链接

7. `许可说明`

每个文档应在末尾包含以下许可声明：

~~~
## 许可声明

本文档采用 [知识共享署名-非商业性使用-相同方式共享 4.0 国际许可协议 (CC BY-NC-SA 4.0)](https://creativecommons.org/licenses/by-nc-sa/4.0/deed.zh) 进行许可。
~~~

代码文件应包含 MIT 许可声明。

## 5.4 写作风格

为确保社区所有文档的一致性、可读性和专业性，所有贡献者在撰写Markdown 文档时，必须遵守以下写作风格规范。

1. `清晰性与简洁性`

- 使用主动语态：强制要求使用主动语态，以增强清晰度和直接性。例如，使用“该脚本处理数据...”而非“数据被该脚本处理...” 。
- 使用简洁句式：提倡使用简短、清晰的句子。避免冗长、复杂的从句结构。推荐使用 Hemingway App 或 Grammarly 等工具进行自我审查，以提高文本的可读性
- 术语一致性：社区应建立并维护一份核心术语表，定义项目中常用的专业术语。所有文档在提及这些术语时，必须使用统一的表达方式，以构建社区的“通用语言”，减少沟通歧义 。

2. `结构化与可读性`

内容结构格式化以利于快速浏览，内容应被分解为简短、易于理解的章节，避免出现大段的说明性文字。善用标题、列表和链接，将信息组织成易于快速导航和阅读的块状结构 。

一篇良好的文档通常包含以下部分：

- 标题 (Title)
- 摘要/前言 (Abstract/Introduction): 简要说明文档的目的、背景和核心结论。
- 正文 (Body): 结构清晰，使用多级标题划分逻辑段落。
- 结论 (Conclusion): 总结要点。
- 参考文献/资源 (References/Resources): 引用所有参考的外部链接、论文或资料，体现专业性并尊重原作者。
- 附录 (Appendix, 可选): 存放额外图表、数据等。

3. `视觉化沟通`

- 展示而非描述：“Show, don't tell”是本社区文档的核心原则之一 。应优先使用视觉元素来传达信息。
- 图表与截图：对于涉及多维度的信息，必须使用表格进行呈现，而非冗长的文字描述 。在解释用户界面、流程或结果时，应附上清晰的截图、GIF动图或架构图。对于命令行工具，可使用开源工具录制操作过程 。
- 图示规范：所有图片和图表都应有编号和简短的标题说明，并在正文中引用。

## 5.5 AI 辅助写作建议

1. 写作时可在文件顶部添加 <!-- llm-prompt: 生成摘要 -->，让 AI 自动补全。
2. 使用Markdown注释保存 prompt：<!-- llm-context: 本文讨论合成劳动力生成权的经济模型 -->
3.  提交前运行 docs/.scripts/llm_lint.py，检查标题层级、缺失元数据。

## 6.  Commit & PR规范

1. `提交信息`

提交信息应遵循以下格式：

- 类型(范围): 简短描述
- 详细描述（可选）
- 相关问题（可选）

类型包括：

- docs: 文档更新
- feat: 新功能
- fix: 错误修复
- refactor: 代码重构
- style: 格式调整
- test: 测试相关

2. `Pull Request`

提交Pull Request应遵循以下格式：

- 标题： 清晰描述PR目的，如 [Docs] Add Eclipse Attack Research。
- 描述：
	- 简要说明本次提交的内容和目的。
	- 关联相关 Issue（如有），例如 Closes #123。
	- 勾选检查项（见下文PR模板）。

Pull Request模板：

~~~
## 变更类型
<!-- 请勾选一项 -->
- [ ] 新研究论文/报告
- [ ] 新教程/指南
- [ ] 文档内容修正/更新
- [ ] 文档结构优化
- [ ] 其他（请注明）________________

## 描述
<!-- 详细描述本次提交的目的和主要内容。 -->

## 相关 Issue
<!-- 关联本次PR旨在解决的Issue，例如 Closes #123 -->
Closes #

## 检查清单 (Checklist)
<!-- 请在提交前确保完成以下事项 -->
- [ ] 我已阅读并同意遵守《文档贡献指南》。
- [ ] 我对文档进行了拼写和语法检查。
- [ ] 我使用了建议的文件命名规则和目录结构。
- [ ] 我的文档结构清晰，包含了摘要、正文和参考文献等必要部分。
- [ ] 我引用了所有参考的资料和来源。
~~~

## 7. 文档审核

所有文档在合并前需经过以下审核流程：

1. **技术审核**：确保技术内容准确
2. **格式审核**：确保符合本指南规定的格式
3. **语言审核**：确保语言表达清晰、准确
4. **一致性审核**：确保与现有文档保持一致

## 文档规范指南总结

遵循此指南将有助于：

- **提高协作效率**： 统一的格式和结构让其他成员更容易阅读、理解和维护文档。
- **保证内容质量**： 通过规范化的流程，确保提交的文档清晰、准确、有价值。
- **维持仓库整洁**： 良好的文件组织方式使仓库结构清晰，便于导航和管理。
- **展现社区专业性**： 统一的规范体现了社区的严谨性和专业性，吸引更多优秀的贡献者。

最后，感谢你花时间阅读本指南并为Gitconomy Research社区做出贡献！你的每一份努力都帮助我们共同构建一个更强大、更专业的开源研究社区。
